# Phase 1A Windows AppHost Supervision Spike Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prove that a .NET 10 Windows AppHost can safely own a real local PostgreSQL 16 cluster plus fixture server and worker processes with atomic containment, authenticated control, deterministic recovery, and bounded diagnostics.

**Architecture:** Add an isolated `desktop/windows` .NET solution beside Twenty's existing TypeScript packages. A platform-neutral lifecycle core emits generation-fenced commands, a Windows implementation owns native Job/process/pipe/profile primitives, and a thin AppHost composes those pieces with a real PostgreSQL runtime and purpose-built fixture children.

**Tech Stack:** .NET 10 LTS, C# 14, `net10.0`, `net10.0-windows`, xUnit generated by the .NET 10 SDK template, Windows Job Objects and named pipes through source-generated `LibraryImport`, PostgreSQL 16 native Windows binaries (16.14 at review time), `psql`, `initdb`, and `pg_ctl`.

## Global Constraints

- Start from base commit `3fda463e`; the approved design is `docs/superpowers/specs/2026-07-14-windows-apphost-supervision-spike-design.md`.
- Use .NET 10. Platform-neutral projects target `net10.0`; Windows projects target `net10.0-windows`.
- Keep all C# source under `desktop/windows`; do not add .NET concerns to Twenty's Nx/Yarn packages.
- Publish an ordinary self-contained `win-x64` folder. Do not enable single-file, trimming, or NativeAOT.
- Production AppHost code must launch absolute executables directly. It must never invoke PowerShell, `cmd.exe`, `npm`, or `npx`.
- Use fixture server and worker processes. Do not claim or test full Redis-free Twenty startup in Phase 1A.
- Use a real native PostgreSQL 16 cluster in disposable test profiles. Never touch `%LOCALAPPDATA%\HowardLab\eBayCRM` from tests.
- Atomically assign every supervised process to one unnamed, non-inheritable `KILL_ON_JOB_CLOSE` Job during `CreateProcessW`.
- Treat every timeout as indeterminate until process, pipe, and PostgreSQL state are reconciled.
- Fence every delayed signal by role, generation, and operation ID; serialize all lifecycle transitions.
- Keep passwords and capability nonces out of process arguments, filenames, manifests, and typed diagnostics.
- Use red-green TDD for each behavior. Run focused tests after each step and commit after every task.
- Use injected short deadlines and fake clocks in timeout tests; do not wait for production 60-second or five-minute bounds.
- Stop implementation and record `REVISE_APPHOST_FOUNDATION` if atomic Job assignment, real-postmaster ownership, or no-duplicate timeout reconciliation cannot be proven.

- [ ] **Step 2: Define the exact state, event, command, and generation types**

Use the approved state names, including `ReconcilingDatabaseStart`, `ReconcilingDatabaseStop`, `Stopping`, and `Faulted`. Define:

```csharp
public readonly record struct ProcessGeneration(
    RuntimeRole Role,
    long Value,
    Guid OperationId);

public abstract record LifecycleEvent(ProcessGeneration? Generation);
public sealed record StartRequested(Guid OperationId) : LifecycleEvent(null);
public sealed record StopRequested(Guid OperationId) : LifecycleEvent(null);
public sealed record RoleReady(ProcessGeneration Value) : LifecycleEvent(Value);
public sealed record RoleExited(ProcessGeneration Value, int ExitCode) : LifecycleEvent(Value);
public sealed record HealthFailed(ProcessGeneration Value, string ReasonCode) : LifecycleEvent(Value);

public sealed record Reconciled(ProcessGeneration Value, ReconciledState State) : LifecycleEvent(Value);
```

`LifecycleCommand` contains only typed, non-secret data: command type, target generation, operation ID, and deadline key. Do not place raw exception messages or environment dictionaries in these records.

- [ ] **Step 3: Implement the single-writer coordinator**

Use one private `SemaphoreSlim(1, 1)` around the transition function. The transition function must:

1. reject stale generations before consulting current state;
2. record the first failure key `(role, generation, recovery-kind)` and ignore duplicates;
3. reject every restart once `StopRequested` is accepted;
4. emit commands rather than performing I/O;
5. increment a role's generation only when it emits a start command; and
6. keep PostgreSQL running for server-only and worker-only recovery.

Return:

```csharp
public sealed record TransitionResult(
    RuntimeState Previous,
    RuntimeState Current,
    IReadOnlyList<LifecycleCommand> Commands,
    bool Ignored,
    string ReasonCode);
```

- [ ] **Step 4: Write failing restart-budget and shutdown-race tests**

Require two retries, a third failure that faults exactly once, a reset after the injected stable period, and no restart after shutdown begins. Use `TestClock` to advance time without sleeping.

- [ ] **Step 5: Implement `RestartBudget` and finish the recovery matrix**

`RestartBudget.TryConsume(RuntimeRole role, DateTimeOffset now)` returns `Allowed`, `Exhausted`, or `ResetAndAllowed`. `RecordStable(RuntimeRole role, DateTimeOffset now)` begins the injected stable interval. Tests configure two retries in sixty seconds and a five-minute reset; production values remain an injected policy decision.

Server failure commands are exactly `StopWorker`, `StartServer`, `WaitForServer`, `StartWorker`, `WaitForWorker`. Worker failure restarts only the worker. Database failure commands stop worker then server before database reconciliation.

### Type consistency

- `RuntimeRole` is defined once in Protocol and consumed by Core, Windows, AppHost, and Fixture.
- `ProcessGeneration` is defined once in Core and is the generation key for lifecycle and Windows identities.
- `ControlEndpointIdentity` maps one-to-one to the protocol `HelloPayload` plus retained process identity.
- `PostgresInstanceIdentity` is returned by `IPostgreSqlRuntime` and is the only identity accepted by migration and stop operations.
- `SecretValue` is the only secret-bearing value accepted by `LaunchSpecification`; diagnostic records cannot accept it.
- AppHost composition consumes interfaces from earlier tasks and contains no duplicate policy or native API declarations.
